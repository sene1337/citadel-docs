Foundry vs OpenClaw vs Citadel OS

A direct comparison of Darryl Hicks' Agent Foundry / Building Brit system against our 500-hour OpenClaw operating layer and the target Citadel OS blueprint.

Executive verdict

Darryl's system is directionally right, but the public repo is not a working OS. It is a strong pattern library: principles, gates, schemas, archetypes, and sanitized examples.

Borrow the gates, schemas, role-spec discipline, and sanitized self-serve repo model. Do not blindly adopt the full agent org chart or assume the public artifacts prove runtime reliability.

Foundry as philosophy

9/10

The principles match our lived scars: durable memory, skills, gates, specialists, and proof over vibes.

Public repo as pattern library

8/10

Clean, useful, self-serve templates. Strong for adaptation, not installation.

Ready-to-run OS

3/10

The messy runtime layer is mostly absent: auth, delivery, compaction, rollback, routing, health checks.

What Darryl's system is

The Agent Foundry is a framework-agnostic companion to Building Brit: a catalog of reusable patterns for founders building an AI operating layer. It includes ranked principles, fail-closed gates, security patterns, skill schemas, agent archetypes, tool specs, and pickup prompts another agent can use.

Core thesis

A chatbot answers. An agent remembers, acts, and compounds. The controllable pieces are personality, memory, skills, and tools.

Specialist model

Brit orchestrates; Victor engineers; Nora processes calls; Dex briefs; Ivy writes; Greg maintains runtime health.

What we've built in OpenClaw / Citadel Ops

Our setup is already a complex OpenClaw operating layer: identity, memory, daily logs, decision ledger, regressions, SOPs, skills, Telegram routing, subagents, project registries, file janitor, lossless recall, and real workflows that survived pressure.

  • Identity/persona files: SOUL, IDENTITY, USER, AGENTS, MEMORY.
  • Durable memory, daily logs, wiki/search, session recall.
  • Large active skill stack with progressive disclosure.
  • First-class tools for browser, messages, cron, nodes, gateway config, GitHub, PDFs, images, wallets, wiki, memory, sessions.
  • Continuous-improvement ledgers for regressions and capability gaps.
  • Subagent delegation with file outputs and parent verification.

What Citadel OS is trying to become

Citadel OS is broader than an agent stack. It is a life-wide operating system for a Bitcoin-focused founder running media, investments, nonprofit/Bitcoin, financial/legal, properties, family, health, and personal development.

Layer 1 · Founder's Life OS

Cross-domain rocks, capital allocation, Chief of Staff seat, domain oversight, replacement ladder.

Layer 2 · Domain Execution

Dan Martell-style rocks, scorecards, weekly rhythm, DRI model, SOPs/playbooks per domain.

Reality check

Proven

  • Skills as scar tissue
  • Hard gates over prompt wishes
  • Specialists for bounded work
  • Sanitized self-serve repo model

Plausible

  • Victor-style engineering loop
  • Call processor / briefer / writer lanes
  • Mechanic agent, if scoped and audited

Fragile

  • Agent-to-agent handoff
  • External communication routing
  • Source integrations and auth
  • Context size and compaction

Aspirational

Foundry as a deployable operating system; a fully autonomous specialist team; memory that “always has the right context.” Those require boring plumbing: source boundaries, delivery gates, auth, write approvals, rollback, queue ownership, health checks, and proof artifacts.

OpenClaw limitations discovered in practice

Rules are not enforcement

Repeated Telegram delivery/topic regressions prove that documented instructions are not enough. Foundry is right: critical controls need plumbing gates.

Context blows up

Long source-heavy work can overflow. Even this job's first large subagent attempt failed from context size. Progressive disclosure matters.

Delivery is separate from work

Writing a result is not the same as Brad receiving it. GitHub Pages and Telegram both require delivery verification.

Auth is real work

OAuth, 1Password, PATs, browser sessions, and credentials become operational bottlenecks.

Subagents need strict DoD

Without output paths and quality gates, delegation produces status theater or chat sludge.

Source-of-truth drift

Memory is useful but must not outrank current files, wikis, logs, and source systems.

Recommended integration path

Borrow

  • Hard vs reasoning gate taxonomy
  • Build manifest schema
  • Security egress test matrix
  • Skill landmines / fixture discipline
  • Victor's six-question design gate
  • Sanitized repo pattern

Avoid

  • Full Brit/Victor/Nora/Dex/Ivy/Greg org chart immediately
  • Treating public artifacts as implementation proof
  • More self-maintenance rituals before scorecards are live
  • Model-authored gates without deterministic tests

Test next

  1. Create a Foundry-inspired OpenClaw gate pack: delivery, source-of-truth, context-size, external-send approval.
  2. Run a Victor-style build manifest on one real OpenClaw improvement.
  3. Build one specialist only: briefer or call processor.
  4. Map Foundry schemas to our skill-building playbook.
  5. Create a public-safe Citadel Ops pattern repo only after sanitization gates are mechanical.
